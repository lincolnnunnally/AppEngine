# Phase Creation Approval Gate

AppEngine must review a `final_packet_materialization` before phase issues can be generated.

This gate produces a `phase_creation_approval` artifact. It is approval-gate only. It does not create phase issues, trigger Codex build work, build UI, deploy production, create paid resources, apply migrations, add secrets, change env vars, change repository visibility, or auto-merge generated app code.

## Purpose

The Phase Creation Approval Gate answers:

- Is the final packet complete enough to create phase issues?
- Does the final packet still align with the source of truth?
- Is the selected solution type correct?
- Is the phase sequence ready?
- Are provider/cost and security/privacy risks still blocked or safe?
- Are owner approval notes recorded?

This is the final safety checkpoint before AppEngine may generate bounded phase issue drafts in a later step. The later generator is defined in `source-of-truth/phase-issue-generation.md` and still must not create live GitHub issues or trigger build work.

## Input Artifact

The input artifact is `final_packet_materialization`.

Required input fields:

- `kind: "final_packet_materialization"`
- `schemaVersion`
- `sourceArtifact.kind: "packet_draft_approval"`
- `sourceArtifact.candidateSlug`
- `sourceArtifact.candidateType`
- `sourceArtifact.approvalStatus`
- `sourceArtifact.approvedDraftKind`
- `candidate.name`
- `candidate.slug`
- `candidate.type`
- `candidate.summary`
- `candidate.needAddressed`
- `candidate.desiredTransformation`
- `finalPacketType`
- `finalPacket.kind`
- `finalPacket.status`
- `decision.materializationStatus`
- `decision.nextSafeAction`
- `decision.finalPacketCreated`
- `decision.phaseIssuesCreated`
- `decision.codexBuildTriggered`
- `decision.ownerApprovalRequired`
- `guardrails.finalPacketCreationOnly`

The gate must fail honestly when required materialization fields are missing.

## Supported Final Packet Types

The approval gate supports:

- `app_build_packet`
- `vnext_packet`
- `non_app_solution_plan`

The gate must fail honestly for unknown final packet types or when `finalPacket.kind` does not match `finalPacketType`.

## Approval Statuses

Every `phase_creation_approval` must use one of these statuses:

- `approved_for_phase_creation`
- `needs_revision`
- `rejected`
- `blocked_by_security`
- `blocked_by_cost`
- `blocked_by_scope`

Only `approved_for_phase_creation` may recommend later phase issue generation. Even then, this gate does not create phase issues.

After this gate approves phase creation, use `source-of-truth/phase-issue-generation.md` to draft bounded phase issues for owner review before any live GitHub issue is created.

## Required Approval Checks

Every approval must check:

- `finalPacketCompleteness`
- `sourceOfTruthAlignment`
- `correctSolutionType`
- `phaseSequenceReadiness`
- `costProviderSafety`
- `securityPrivacySafety`
- `ownerApprovalNotes`

Each check must include:

- `status`
- `notes`

Allowed check statuses are:

- `pass`
- `needs_revision`
- `rejected`
- `blocked_by_security`
- `blocked_by_cost`
- `blocked_by_scope`

Missing check status or notes must fail honestly.

## Owner-Readable Approval Output

The owner-readable output should be concise:

```text
Phase Creation Approval

Candidate: Church Care Follow-Up
Final packet type: non_app_solution_plan
Status: approved_for_phase_creation
Next safe action: prepare_phase_issue_generation
Owner approval recorded: yes, phase issues may be generated by a later explicit generator
Guardrails: approval gate only, no phase issues, no build, no deploy
```

When the gate does not approve phase creation, the output must clearly say why and what the next safe action is.

## Decision Rules

Approval status is determined from the required approval checks:

- Any `blocked_by_security` check makes the artifact `blocked_by_security`.
- Any `blocked_by_cost` check makes the artifact `blocked_by_cost`.
- Any `blocked_by_scope` check makes the artifact `blocked_by_scope`.
- Any `rejected` check makes the artifact `rejected`.
- Any `needs_revision` check makes the artifact `needs_revision`.
- All checks must be `pass` to produce `approved_for_phase_creation`.

## Guardrails

Phase creation approval must not:

- build UI
- create phase issues
- trigger Codex build work
- deploy production
- create paid resources
- apply migrations
- add secrets or env vars
- change repository visibility
- auto-merge generated app code
- treat phase creation approval as implementation approval

## Success Criteria

The gate is working when:

1. Final App Build Packets can be approved for later phase creation without creating phase issues.
2. Final vNext Packets can be approved for later phase creation without creating phase issues.
3. Final Non-App Solution Plans can be approved for later phase creation without creating phase issues.
4. `needs_revision`, `rejected`, `blocked_by_security`, `blocked_by_cost`, and `blocked_by_scope` are represented honestly.
5. Missing materialization or approval fields fail honestly.
6. The owner can see the source candidate, final packet type, approval status, blockers, guardrails, and next safe action.
7. No phase issues, build work, deployments, migrations, paid resources, secrets, env changes, repository visibility changes, or auto-merges occur.
