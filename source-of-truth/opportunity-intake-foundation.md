# Opportunity Intake Foundation

## Purpose

Opportunity Intake is the customer-facing problem-to-solution front door.

It helps a person, leader, team, or community describe a problem, name who is affected, clarify what better could look like, and route toward the right next solution path.

## Boundary

Opportunity is not the ecosystem itself.

Opportunity is not Spark of Hope, Live On Mission, Best Life, ChurchConnect, Kindred Connection, or any other destination app or service.

Opportunity is the guided entry point that helps AppEngine understand whether the next step should be an app, tool, workflow, content/resource, community/ministry model, existing ecosystem service later, or more clarification.

## AppEngine Role

AppEngine is the production tool and factory.

Opportunity Intake may create reviewable intake records for AppEngine, but it must not automatically create GitHub issues, apply labels, trigger Codex, deploy production, create paid resources, run migrations, add secrets, or change repository visibility.

## Ecosystem Role

ecosystem apps and services are possible destinations or solution components.

Opportunity Intake must not claim a destination exists, is complete, or can serve a user until AppEngine has verified that destination through the proper build, review, release, and monitoring gates.

## Product Doctrine

Transformation and problem-solving are the product.

Apps, websites, workflows, content, services, communities, and automations are tools that help remove barriers and move people toward life.

People are the purpose. Opportunity Intake exists to help people move from pain, confusion, or desire to clarity, useful action, and a safe next step.

## Required Capture

Every Opportunity Intake record should capture:

- problem or pain
- who is affected
- what better outcome looks like
- current barriers
- existing idea or vision, if any
- desired impact
- possible solution type

## Routing Outcomes

Opportunity Intake may route toward:

- app, tool, or workflow need
- content or resource need
- community or ministry model need
- existing ecosystem service later
- needs clarification

## Owner Review

Owner review is required before an Opportunity Intake becomes:

- a `problem_solution_intake`
- a portfolio candidate
- a packet draft
- a GitHub issue
- a Codex handoff
- implementation work

## Guardrails

- Production deploys remain blocked.
- Paid resources remain blocked.
- Live migrations remain blocked.
- Secrets and env changes remain blocked.
- Repository visibility changes remain blocked.
- Codex auto-execution remains blocked.
- GitHub issue creation remains blocked.
- Label changes remain blocked.
- Ecosystem destinations are not assumed to be built.
