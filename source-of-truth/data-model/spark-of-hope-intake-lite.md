## Scope

This phase defines a schema plan for the generated app. It does not create migrations, seed data, API routes, UI routes, Neon branches, provider resources, production deployment settings, or real user data.

The model is app-scoped and separate from the full Spark of Hope product. Any later integration with the full Spark of Hope product requires explicit consent, a documented integration reason, and a new packet or scoped follow-up issue.

## Naming And Persistence Assumptions

- Database provider: Neon planning only, using a generated-app branch or app-scoped database in a later approved phase.
- Database engine: Postgres-compatible schema.
- Table prefix: `sohil_` for Spark of Hope Intake Lite generated-app tables.
- Primary keys: UUID values generated by the database or application in a later implementation phase.
- Timestamps: `created_at` and `updated_at` on primary workflow tables.
- Soft deletion: use `deleted_at` and `deleted_by_membership_id` where record history is needed.
- No seed data with real people is allowed. Later test fixtures must use synthetic names, emails, phone numbers, and story text.

## Access Boundary Matrix

| Role or state | Story body | Contact details | Review notes | Response draft | Audit events | Memberships/settings |
| --- | --- | --- | --- | --- | --- | --- |
| `public_story_sharer` | Create only | Create only | No access | No access | No access | No access |
| `customer` | Own story only if account status surfaces are enabled | Own contact details | No access by default | Approved own response only if exposed later | No access | Own account only |
| `encouragement_volunteer` | Assigned minimum context only | No access by default | No access by default | Assigned response only | No access | No access |
| `reviewer` | Assigned or authorized stories | Assigned contact details only when needed | Assigned review records | Assigned responses | No access by default | No access |
| `admin` | App queue | App contact records for support and response coordination | App review records | App response records | App audit records | Manage app memberships/settings |
| `owner` | App queue | App contact records for support and response coordination | App review records | App response records | App audit records | Manage app, release, and high-risk actions |

Server-side checks are required for every protected read, mutation, export, deletion, assignment, status transition, membership change, and Super Admin action.

## Retention, Deletion, And Export Notes

- Default retention should be limited and app-specific. A later build or release phase must choose the exact retention window before production.
- Story submissions should be exportable for the story sharer or authorized owner/admin review, with audit events for export request and completion.
- Deletion should soft-delete workflow records first and redact story/contact content where legally and operationally appropriate.
- Audit events may be retained after story deletion when needed to prove administrative action, but they must not contain story body or contact values.
- Contact permission withdrawal should prevent future contact use and hide contact data from non-owner/admin users.
- Public share permission is false by default and does not create publication rights without a later approved public-sharing workflow.
- Exports must exclude secrets, internal audit metadata that is not appropriate for sharers, unrelated user data, and data from other apps.

## Privacy And Logging Notes

- Do not log `story_body`, `response_body`, contact email, contact phone, raw request payloads, tokens, provider credentials, database URLs, or session secrets.
- Health, logs, deployment status, and Super Admin registry/status surfaces must contain operational metadata only.
- Story records must not be reused by ChurchConnect, Live On Mission, United Under God, Kids Need Dads, Toner Management, the full Spark of Hope product, or any unrelated app without documented consent and integration approval.
- Volunteers should see the minimum necessary story context and should not receive contact data or broad review notes by default.

## Migration And Rollback Notes

- No migration is applied in this phase.
- A later migration phase must create reviewable SQL or framework migration files and run them only after an approved build task authorizes it.
- Every migration must include rollback notes before preview or production use.
- Destructive changes require a restore point or backup strategy before production.
- Schema changes that affect story privacy, deletion, exports, contact data, role grants, or audit events require release-gate review before production.
- Rollback should preserve auditability where possible while preventing accidentally restored private story/contact content from becoming visible.

## Acceptance Criteria

- Tables or model objects are defined for story submissions, consent/contact, review assignments, encouragement responses, audit/status events, app-scoped memberships, roles, and permissions.
- Ownership, retention, deletion, privacy, export, audit, and rollback notes are explicit.
- Story data stays private by default and separate from the full Spark of Hope product.
- Reviewer and volunteer access boundaries are explicit.
- No migration is applied, no Neon resource is provisioned, no seed data is created, no UI/API route is implemented, and no real user data is created.

Proceed to the Identity/Auth phase to turn these roles, permissions, and access boundaries into a protected route/API matrix. Keep implementation, migrations, and seed data blocked until a later scoped MVP build issue explicitly authorizes them.
